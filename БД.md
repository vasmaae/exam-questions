# **1) Концепция баз данных** 
Концепция баз данных заключается в следующем:
    • Данные должны быть организованы в базы данных с целью адекватного отражения реального объекта и обеспечения информационных нужд пользователей. База данных - это поименованная и структурированная совокупность взаимосвязанных данных, отражающих состояние объектов определенной предметной области и управляемых центральной программной системой.
    • Основная особенность систем управления базами данных (СУБД) - наличие процедур для ввода, хранения и управления не только самими данными, но и их структурой. Файлы, описывающие структуру хранимых данных и управляемые СУБД, стали называться базами данных.
    • Концепция баз данных позволяет достичь таких свойств, как: многоразовое использование данных, простота обновления, быстрый поиск информации, уменьшение избыточности, защита от несанкционированного доступа и повреждения данных, максимальная независимость от прикладных программ.
    • Основными требованиями к базам данных являются: адекватное отражение предметной области, полнота данных, динамичность.
В целом, концепция баз данных направлена на централизованное хранение, управление и использование данных, что значительно повышает эффективность информационных систем.
Идея повышения степени независимости обрабатывающих программ от способов хранения и содержания хранимых данных впервые была использована в концепции баз данных путем разделения логического и физического уровней хранения данных в 1964 году в исследованиях сотрудников фирмы IBM.

# **2) Системы управления базами данных** 
Системы управления базами данных (СУБД) - это комплекс программных средств, которые позволяют создавать, управлять и обрабатывать базы данных. Основные функции СУБД:
    • Создание, редактирование и удаление данных в базе данных
    • Обеспечение безопасности и целостности данных
    • Предоставление средств для доступа, поиска и извлечения данных
    • Поддержка разных типов баз данных, таких как реляционные, NoSQL, распределенные и другие
Ведущие СУБД включают PostgreSQL, MySQL, Microsoft SQL Server, SQLite, MongoDB, Oracle Database, DB2 и Redis. Они используются в самых разных приложениях и сферах, от веб-сайтов до крупных корпоративных систем. СУБД служат связующим звеном между приложениями и базами данных, обеспечивая эффективное управление информацией.

# **3) Общие принципы классификации СУБД** 
Принципы классификации систем управления базами данных (СУБД) можно разделить на несколько основных критериев:
По степени универсальности
    • СУБД общего назначения - не ориентированы на какую-либо конкретную предметную область
    • СУБД специального назначения - ориентированы на определенную предметную область
По модели данных
    • Иерархические СУБД - данные представлены в виде древовидной структуры
    • Сетевые СУБД - расширение иерархической модели, данные могут иметь несколько предков
    • Реляционные СУБД - данные организованы в виде связанных таблиц
    • Объектно-ориентированные СУБД - данные представлены в виде объектов
    • Объектно-реляционные СУБД - объединяют концепции реляционной и объектной моделей
По степени распределенности
    • Локальные СУБД - все компоненты размещаются на одном компьютере
    • Распределенные СУБД - компоненты могут размещаться на двух и более компьютерах
По способу доступа к БД
    • файл-серверные СУБД - файлы баз данных хранятся на сервере
    • клиент-серверные СУБД - компоненты СУБД распределены между клиентом и сервером
Таким образом, СУБД можно классифицировать по нескольким основным признакам, что определяет их функциональные возможности и сферу применения.

# **4) Определение предметной области, сущности, атрибута и кортежа**  
Предметная область - это часть окружающего мира, которую мы рассматриваем в рамках базы данных. Она включает в себя объекты, события, понятия, которые необходимо описать и отразить в структуре базы данных.
Сущность - это объект предметной области, который мы хотим представить в базе данных. Сущности обладают характеристиками, называемыми атрибутами.
Атрибут - это отдельная характеристика сущности, которая имеет определенное название и тип данных. Атрибуты описывают свойства объектов предметной области.
Кортеж - это совокупность значений атрибутов, которая однозначно описывает один экземпляр сущности. Набор кортежей, описывающих объекты одного класса, называется отношением.

# **5) Понятие отношения и формы представления отношений**
Основополагающим концептом реляционной модели данных является понятие "отношение". Отношение - это множество кортежей (записей), которые соответствуют одной схеме отношения. Схема отношения - это набор пар "имя атрибута, тип данных".
Отношение можно представить в нескольких формах:
    1. Таблица - данные отображаются в виде двухмерной таблицы, где строки - кортежи, а столбцы - атрибуты.
    2. Множество - отношение рассматривается как математическое множество кортежей, без какого-либо упорядочивания.
    3. Список - отношение представляется в виде неупорядоченного списка кортежей.
    4. Наибольшее распространение получило представление отношений в виде графических диаграмм, например ER-диаграмм.

# **6) Нормальные формы и процесс нормализации**
Нормальные формы (НФ) - это набор правил и требований, которые используются для организации данных в реляционных базах данных с целью минимизации избыточности данных и повышения целостности. Процесс нормализации заключается в поэтапном преобразовании структуры базы данных для приведения её к одной из нормальных форм. Нормализация отношений информационной модели предметной области является механизмом создания логической модели реляционной базы данных. 
Основные этапы процесса нормализации:
    1. Первая нормальная форма (1НФ): устранение повторяющихся групп в отдельных таблицах. Отношение находится в первой нормальной форме (1НФ), если все атрибуты отношения являются простыми (требование атомарности атрибутов в реляционной модели), т.е. не имеют компонентов. 
    2. Вторая нормальная форма (2НФ): создание отдельных таблиц для наборов взаимосвязанных данных. Отношение находится во второй нормальной форме (2НФ), если оно находится в 1НФ, и все неключевые атрибуты отношения функционально полно зависят от составного ключа отношения. 
    3. Третья нормальная форма (3НФ): устранение транзитивных зависимостей, когда одно поле зависит от другого, не являющегося ключевым. Отношение находится в третьей нормальной форме (3НФ), если оно находится во 2НФ, и все неключевые атрибуты отношения зависят только от первичного ключа. 
1НФ - все атрибуты отношения простые;  2НФ - отношение находится в 1НФ и не содержит частичных ФЗ;  3НФ - отношение находится во 2НФ и не содержит транзитивных ФЗ от ключа;  НФБК - отношение находится в 3НФ и не содержит ФЗ ключей от неключевых атрибутов;  4НФ, применяется при наличии более чем одной многозначной ФЗ - отношение находится в НФБК или 3НФ и не содержит независимых многозначных ФЗ;  5НФ - отношение находится в 4НФ и не содержит ФЗ по соединению.

# **7) Связи: виды и определения**
«Один к одному» – когда одной строке из одной таблицы соответствует одна строка из другой таблицы 
«Один ко многим» – когда одной строке из одной таблицы соответствует несколько строк из другой таблицы 
«Многие к одному» - когда нескольким строкам из одной таблицы соответствует одна строка из другой таблицы 
«Многие ко многим» - когда нескольким строкам из одной таблицы соответствует несколько строк из другой таблицы

# **8) Создание схемы данных** 
Создание схемы данных является важным этапом при проектировании базы данных. Схема данных определяет структуру и взаимосвязи между таблицами, полями и другими объектами базы данных. Она помогает визуализировать и организовать данные, что упрощает разработку, управление и взаимодействие с базой данных.
Основные шаги для создания схемы данных в базе данных:
    1. Определение требований к данным: Изучите потребности пользователей и бизнес-требования, чтобы понять, какие данные необходимо хранить и как они будут использоваться.
    2. Моделирование концептуальной схемы: На этом этапе вы определяете основные сущности (таблицы), их атрибуты (поля) и связи между ними. Это поможет вам визуализировать структуру данных.
    3. Проектирование логической схемы: Преобразуйте концептуальную схему в логическую модель, определяя типы данных, ключи, индексы и другие атрибуты таблиц.
    4. Реализация физической схемы: Воплотите логическую схему в конкретную базу данных, создав таблицы, связи и другие объекты. Это может включать выбор СУБД, создание SQL-скриптов и т.д.
    5. Тестирование и оптимизация: Проверьте работоспособность схемы, выполняя тестовые запросы и оценивая производительность. При необходимости внесите изменения в структуру данных.
Создание качественной схемы данных требует тщательного планирования и итеративной работы. Это ключевой шаг для построения эффективной и масштабируемой базы данных.

# **9) Ограничения** 
Ограничения в базах данных (БД) - это правила, которые гарантируют целостность и достоверность данных. Они помогают обеспечить согласованность данных в таблицах и между ними.
Основные типы ограничений в БД:
    1. Первичный ключ (PRIMARY KEY): Каждая таблица должна иметь первичный ключ, который однозначно идентифицирует каждую строку в таблице.
    2. Внешний ключ (FOREIGN KEY): Ограничение, связывающее колонку с данными из другой таблицы. Это гарантирует целостность данных.
    3. Уникальность (UNIQUE): Ограничение, которое позволяет хранить только уникальные значения в колонке.
    4. Проверка (CHECK): Ограничение, которое проверяет, что значение в колонке соответствует определенному условию.
    5. Обязательность (NOT NULL): Ограничение, которое требует, чтобы значение в колонке не было NULL.
    6. По умолчанию (DEFAULT): Ограничение, которое устанавливает значение по умолчанию, если значение для колонки не указано.
